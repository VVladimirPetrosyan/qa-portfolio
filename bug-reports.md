## BUG-001: Публикация отложенного поста с прошедшим временем не показывает ошибку

**Severity:** Medium  
**Priority:** P2  
**Status:** Open  
**Module:** Планировщик отложенных постов  
**Тип:** Defect

### Шаги для воспроизведения:
1. Открыть приложение `http://localhost:5000`
2. Ввести ссылку на товар: `https://example.com/product`
3. Нажать «Сгенерировать»
4. В поле «Время публикации» ввести дату 1 января 2020 года
5. Нажать «Запланировать»

### Ожидаемый результат:
Сообщение об ошибке: «Время публикации уже прошло»

### Фактический результат:
Пост добавляется в список запланированных и никогда не публикуется (висит вечно)

### Стек вызова:
```python
# В check_scheduled_posts():
publish_at = datetime.fromisoformat(post["publish_at"])
if publish_at > now:
    remaining.append(post)
    continue
# Пост остаётся в remaining навсегда
```

### Причина:
Функция `check_scheduled_posts()` не удаляет посты с прошедшим временем, а просто оставляет их в `remaining`

### Рекомендация:
Добавить проверку перед добавлением в scheduled: `if publish_at <= now: return error`

## BUG-002: Retry-логика не сбрасывает счётчик при успешной публикации

**Severity:** Medium  
**Priority:** P2  
**Status:** Open  
**Module:** Retry-логика  
**Тип:** Defect

### Шаги для воспроизведения:
1. Эмулировать ошибку VK API (невалидный токен)
2. Опубликовать пост → ошибка → добавляется в retry
3. Исправить токен
4. Дождаться успешной повторной попытки
5. Проверить `retry.json`

### Ожидаемый результат:
Пост удаляется из retry после успешной публикации

### Фактический результат:
Пост остаётся в retry со `maxed_out: false` и `retry_count` не сброшен

### Стек вызова:
```python
# В check_scheduled_posts():
if post_id:
    # Пост опубликован, но НЕ удаляется из new_retries
    retry_changed = True
    # ... добавляется в published
    # НО: new_retries.append(r) — пост остаётся в списке
```

### Причина:
В цикле retry при успешной публикации пост добавляется в `new_retries` вместо того, чтобы быть пропущенным

### Рекомендация:
Не добавлять успешный пост в `new_retries`:
```python
if post_id:
    # Успешно — не добавляем в new_retries
    published.append(...)
else:
    # Не успешно — обновляем и добавляем
    new_retries.append(r)
```

## BUG-003: Генерация поста с пустой ссылкой не показывает ошибку при наличии изображений

**Severity:** Low  
**Priority:** P3  
**Status:** Open  
**Module:** Генерация поста  
**Тип:** Defect

### Шаги для воспроизведения:
1. Открыть приложение
2. Ввести ручные URL изображений (без ссылки на товар)
3. Оставить поле «Ссылка на товар» пустым
4. Нажать «Сгенерировать»

### Ожидаемый результат:
Сообщение: «Введите ссылку на товар»

### Фактический результат:
Поле ошибки не отображается, но пост не генерируется (тихий фейл)

### Причина:
Проверка на пустую ссылку стоит в `if not product_link`, но поле `image_urls` обрабатывается до этой проверки

### Рекомендация:
Вынести проверку ссылки в самое начало обработки формы

## BUG-004: Кодировка кириллицы ломается в отчёте Allure при запуске через GitHub Actions

**Severity:** Medium  
**Priority:** P2  
**Status:** Open  
**Module:** CI/CD  
**Тип:** Environment

### Шаги для воспроизведения:
1. Запустить тесты локально → Allure-отчёт корректный
2. Запустить тесты через GitHub Actions
3. Открыть развёрнутый Allure-отчёт

### Ожидаемый результат:
Кириллица отображается корректно

### Фактический результат:
Кириллица отображается как `???` или `Ð°Ð±Ð²`

### Причина:
В GitHub Actions Ubuntu по умолчанию использует локаль `C` без поддержки UTF-8

### Рекомендация:
Добавить в workflow:
```yaml
env:
  JAVA_OPTS: "-Dfile.encoding=UTF-8"
  LANG: "ru_RU.UTF-8"
```

## BUG-005: XSS-уязвимость в поле ручного ввода URL изображения

**Severity:** High  
**Priority:** P1  
**Status:** Open  
**Module:** Парсинг изображений  
**Тип:** Security

### Шаги для воспроизведения:
1. Открыть приложение
2. В поле «URL изображений» ввести: `javascript:alert('XSS')`
3. Нажать «Добавить»
4. Кликнуть по добавленному «изображению»

### Ожидаемый результат:
URL не добавляется или экранируется

### Фактический результат:
URL добавляется в список, при клике может выполниться скрипт

### Причина:
Нет валидации URL на предмет `javascript:`, `data:` и других опасных протоколов

### Рекомендация:
Добавить валидацию:
```python
ALLOWED_SCHEMES = ("http://", "https://")
if not url.startswith(ALLOWED_SCHEMES):
    return error
```

## BUG-006: Одновременное добавление и удаление из избранного вызывает гонку данных

**Severity:** Low  
**Priority:** P3  
**Status:** Open  
**Module:** Избранное  
**Тип:** Concurrency

### Шаги для воспроизведения:
1. Быстро нажать «В избранное» и сразу «Удалить» для одного поста
2. Проверить `favorites.json`

### Ожидаемый результат:
Последнее действие побеждает

### Фактический результат:
Иногда файл повреждается или остаётся в неконсистентном состоянии

### Причина:
Нет блокировки файла при записи (Race condition)

### Рекомендация:
Использовать файловую блокировку или `fcntl.lockf`
